## Обзор и сравнение

| Алгоритм | Best | Average | Worst | Memory | Stable | Применение |
|----------|------|---------|-------|--------|--------|-----------|
| Bubble | O(n) | O(n²) | O(n²) | O(1) | ✅ | Учебный |
| Insertion | O(n) | O(n²) | O(n²) | O(1) | ✅ | Малые / почти отсортированные массивы |
| Selection | O(n²) | O(n²) | O(n²) | O(1) | ❌ | Минимум swap'ов |
| Merge | O(n log n) | O(n log n) | O(n log n) | O(n) | ✅ | Гарантия, linked lists, внешняя сортировка |
| Quick | O(n log n) | O(n log n) | O(n²) | O(log n) | ❌ | Общего назначения, кэш-дружественный |
| Heap | O(n log n) | O(n log n) | O(n log n) | O(1) | ❌ | In-place, гарантированный O(n log n) |
| Counting | O(n+k) | O(n+k) | O(n+k) | O(k) | ✅ | Целые числа в известном диапазоне |
| Radix | O(d·n) | O(d·n) | O(d·n) | O(n+k) | ✅ | Целые / строки фиксированной длины |
| Tim | O(n) | O(n log n) | O(n log n) | O(n) | ✅ | JavaScript `.sort()` — гибрид merge+insertion |

> **k** — диапазон значений, **d** — количество цифр

**Практический выбор:**
- Общий случай → Quick Sort (кэш-дружественный, хорошие константы)
- Нужна стабильность → Merge Sort или Tim Sort
- Много маленьких подмассивов → Insertion Sort
- Целые числа в небольшом диапазоне → Counting / Radix
- В V8 используется Tim Sort (`.sort()`)

## Bubble Sort

**Идея:** Сравниваем соседей, большие «всплывают» к концу. O(n²).

```
[5, 1, 4, 2]
 ↓
[1, 4, 2, 5]  — 5 «всплыл»
[1, 2, 4, 5]  — 4 «всплыл»
[1, 2, 4, 5]  — уже отсортировано
```

**Оптимизация:** Если за проход не было ни одного swap — массив отсортирован, выходим.

---

## Insertion Sort

**Идея:** Берём следующий элемент и вставляем его на правильную позицию в уже отсортированную часть.

Оптимален для:
- Небольших массивов (< 16-32 элементов)
- Почти отсортированных данных → O(n)
- Онлайн-сортировки (элементы поступают по одному)

Tim Sort использует Insertion Sort для подмассивов размером < 32 (run).

---

## Merge Sort

**Идея:** Divide and Conquer. Делим пополам рекурсивно → сортируем половины → сливаем (merge).

```
[5, 1, 4, 2, 8, 3]
   ↙           ↘
[5, 1, 4]   [2, 8, 3]
  ↙   ↘      ↙   ↘
[5] [1,4] [2,8] [3]
     ↓       ↓
  [1,4]   [2,8]
       ↘ ↙
   [1,2,4,8] → merge с [3,5] → [1,2,3,4,5,8]
```

**Главное:** merge — это O(n) при условии что обе половины отсортированы. Два указателя.

---

## Quick Sort

**Идея:** Выбираем pivot. Всё < pivot → левая часть. Всё > pivot → правая. Рекурсивно.

**Выбор pivot критичен:**
- Первый/последний → O(n²) на отсортированных данных
- Случайный → average O(n log n) с высокой вероятностью
- Median-of-3 → хорошая эвристика

**In-place partitioning (Lomuto / Hoare):**
- Lomuto: проще, больше swap'ов
- Hoare: меньше swap'ов, сложнее

---

## Heap Sort

**Идея:** Строим max-heap за O(n). Каждый раз извлекаем максимум (корень) и помещаем в конец.

**Heap свойство:** Родитель ≥ детей.
- `parent(i) = Math.floor((i-1)/2)`
- `left(i) = 2*i+1`
- `right(i) = 2*i+2`

Гарантированный O(n log n) при O(1) доп. памяти — лучше чем Quick Sort в worst case, хуже по константам.

---

## Counting Sort

**Идея:** Не сравниваем — считаем количество каждого значения. Работает только с целыми числами в известном диапазоне.

O(n + k) где k — диапазон. Если k >> n — нецелесообразно.

---

## Radix Sort

**Идея:** Сортируем поцифрово (от младшего к старшему разряду — LSD). На каждом проходе используем стабильный Counting Sort.

O(d × n) где d — количество цифр/символов.

---

## Tim Sort

Что реально происходит в `Array.prototype.sort()`:
1. Массив разбивается на **runs** (уже отсортированные или почти отсортированные куски)
2. Каждый run доводится до минимального размера (~32) через **Insertion Sort**
3. Runs сливаются через **Merge** с оптимизациями (galloping mode)

Поэтому `.sort()` работает за **O(n)** на уже отсортированных данных.

## Вопросы на интервью

1. **Почему Quick Sort быстрее Merge Sort на практике при одинаковой сложности O(n log n)?**
   > Quick Sort кэш-дружественный: работает in-place, данные в L1/L2 кэше. Merge Sort требует O(n) доп. памяти и случайный доступ при merge. Константа Quick Sort меньше. Но worst case O(n²) — поэтому интросортировка (Quick + Heap при деградации) лучше.

2. **Когда сортировка быстрее O(n log n)?**
   > Когда данные не сравниваются. Counting Sort — O(n+k) для целых в диапазоне. Radix Sort — O(d·n). Нижняя граница O(n log n) доказана только для сортировок основанных на сравнении.

3. **Что такое стабильная сортировка?**
   > Сохраняет относительный порядок равных элементов. Bubble, Insertion, Merge — стабильные. Quick, Heap, Selection — нет. Важно когда сортируем по нескольким полям: сначала по дате, потом по имени — если по имени стабильная, порядок по дате сохранится.

4. **Как `.sort()` работает в V8?**
   > Tim Sort с Node.js 11+. До этого — Quick Sort. Tim Sort гарантирует O(n log n) worst case, стабилен, O(n) для уже отсортированных.

5. **Как реализовать Quick Sort без риска O(n²)?**
   > Выбирать pivot случайно (`random pivot`). Или использовать median-of-3. Или интросортировку: при деградации (глубина рекурсии > 2·log n) переключаться на Heap Sort.
