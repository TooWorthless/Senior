## Зачем семантика на Senior-уровне

Семантика — это не «правильные теги ради красоты». На production это влияет на:

- **SEO**: поисковики используют семантику для понимания структуры страницы и извлечения featured snippets.
- **Accessibility**: screen readers используют семантику для навигации (landmarks, headings tree).
- **Производительность**: браузер применяет встроенные стили и поведение без лишнего JS.
- **Maintainability**: семантичный код читается и рефакторится проще.

> 💬 **Вопрос на интервью:** «Зачем использовать семантические теги, если визуально можно добиться того же с `<div>`?»

**Ответ (concise):** `<div>` не несёт смысла — ни для браузера, ни для AT (Assistive Technologies), ни для поисковика. Семантический элемент имеет встроенную роль в Accessibility Tree, влияет на SEO-ранжирование и создаёт implicit ARIA roles без дополнительных атрибутов.

## Семантические элементы HTML5

### Структурные / Landmark элементы

| Элемент | ARIA Role (implicit) | Когда использовать |
|---------|---------------------|-------------------|
| `<header>` | `banner` (если в `<body>`), `generic` (если в `<article>`/`<section>`) | Шапка страницы или секции |
| `<footer>` | `contentinfo` (если в `<body>`), `generic` (если вложен) | Подвал страницы или секции |
| `<main>` | `main` | Основной контент страницы. **Один на страницу** |
| `<nav>` | `navigation` | Блок навигационных ссылок |
| `<aside>` | `complementary` | Контент, косвенно связанный с основным |
| `<section>` | `region` (если есть accessible name) | Тематическая группировка контента |
| `<article>` | `article` | Самодостаточный контент (пост, карточка, комментарий) |

> ⚠️ **Ловушка:** `<section>` без `aria-label` или `<h*>` внутри не создаёт `region` landmark — screen reader её проигнорирует как структурный элемент. Это частая ошибка.

### Текстовые семантические элементы

| Элемент | Семантика | Частая ошибка |
|---------|-----------|---------------|
| `<strong>` | Важность (importance) | Использовать для визуального bold вместо CSS |
| `<em>` | Ударение/акцент (stress emphasis) | Путать с `<i>` |
| `<b>` | Привлечение внимания без важности | Использовать вместо `<strong>` |
| `<i>` | Альтернативный голос, технический термин, иностранное слово | Использовать для курсива вместо CSS |
| `<mark>` | Подсветка в контексте поиска/цитирования | — |
| `<time>` | Машиночитаемые дата/время | Не ставить атрибут `datetime` |
| `<abbr>` | Аббревиатура с расшифровкой | — |
| `<cite>` | Ссылка на источник/произведение | Использовать для имени автора |
| `<q>` | Inline-цитата | `<blockquote>` — для блочных цитат |
| `<code>` | Код | `<pre><code>` — для блоков кода с сохранением форматирования |
| `<kbd>` | Пользовательский ввод (клавиши) | — |
| `<samp>` | Вывод программы | — |
| `<var>` | Переменная | — |
| `<del>` / `<ins>` | Удалённый / добавленный контент | — |
| `<sup>` / `<sub>` | Надстрочный / подстрочный | — |

### Группировка контента

| Элемент | Семантика |
|---------|-----------|
| `<figure>` + `<figcaption>` | Самодостаточный медиа-контент с подписью |
| `<details>` + `<summary>` | Раскрываемый блок (нативный accordion) |
| `<dialog>` | Нативный модальный/немодальный диалог |
| `<address>` | Контактная информация автора/владельца |

## Document Outline Algorithm

### Старый алгоритм (до HTML5.2)

HTML5 ввёл концепцию «sectioning elements» (`<article>`, `<section>`, `<aside>`, `<nav>`), которые должны были создавать собственный outline. Теоретически это позволяло использовать `<h1>` в каждой секции:

```html
<h1>Статья</h1>
<section>
  <h1>Раздел 1</h1>  <!-- теоретически h2 в outline -->
</section>
```

> ⚠️ **Ловушка:** Этот алгоритм **никогда не был реализован** ни в одном браузере и screen reader. Спецификация была отозвана. На практике heading levels всегда интерпретируются буквально (h1 > h2 > h3...).

### Актуальный подход (WHATWG)

Спецификация WHATWG убрала outline algorithm. Правила:

1. **Один `<h1>` на страницу** — главный заголовок.
2. Иерархия `h1 → h2 → h3 → ...` должна быть логической, без пропусков уровней.
3. `<section>` / `<article>` не «сбрасывают» счётчик заголовков.

```html
<!-- Правильно -->
<main>
  <h1>Блог</h1>
  <article>
    <h2>Пост 1</h2>
    <section>
      <h3>Подраздел</h3>
    </section>
  </article>
</main>
```

## Выбор правильного элемента: частые ошибки

### `<article>` vs `<section>` vs `<div>`

**Правило большого пальца:**
- `<article>` — если контент можно вырвать из контекста и он останется понятным (пост, карточка товара, виджет погоды, комментарий).
- `<section>` — тематическая группировка внутри документа, которая должна быть в оглавлении.
- `<div>` — чисто для стилизации/скриптов, без семантического значения.

> 💬 **Вопрос на интервью:** «Когда `<article>` может быть вложен в `<article>`?»

**Ответ:** Когда вложенный контент сам по себе самодостаточен и связан с родительским. Классический пример — комментарии к посту: `<article>` (пост) содержит `<article>` (каждый комментарий).

### `<header>` и `<footer>` — не только для страницы

Оба элемента могут использоваться внутри `<article>` и `<section>`. Внутри них они не получают роли `banner`/`contentinfo`:

```html
<article>
  <header>
    <h2>Заголовок поста</h2>
    <time datetime="2025-01-15">15 января 2025</time>
  </header>
  <p>Контент...</p>
  <footer>
    <address>Автор: <a href="/author">Иван</a></address>
  </footer>
</article>
```

### `<nav>` — не для каждого списка ссылок

`<nav>` предназначен для **основных блоков навигации**. Не каждая группа ссылок должна быть `<nav>`:

```html
<!-- Правильно: основная навигация сайта -->
<nav aria-label="Основная навигация">...</nav>

<!-- Правильно: хлебные крошки -->
<nav aria-label="Хлебные крошки">...</nav>

<!-- Неправильно: набор ссылок в футере (лучше просто <ul>) -->
<nav>
  <a href="/privacy">Политика</a>
  <a href="/terms">Условия</a>
</nav>
```

> ⚠️ **Ловушка:** Если на странице несколько `<nav>`, каждый **должен** иметь уникальный `aria-label` или `aria-labelledby`, иначе screen reader объявит их все как «navigation» без разбора.

## Микроразметка

Микроразметка позволяет поисковикам (и другим парсерам) извлекать структурированные данные из HTML.

### JSON-LD (рекомендовано Google)

Встраивается в `<script type="application/ld+json">`. Не смешивается с разметкой — легко обновлять.

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Как работает Shadow DOM",
  "author": {
    "@type": "Person",
    "name": "Иван Петров"
  },
  "datePublished": "2025-01-15",
  "image": "https://example.com/og-image.jpg"
}
</script>
```

### Microdata (встроен в HTML)

```html
<article itemscope itemtype="https://schema.org/Article">
  <h1 itemprop="headline">Заголовок</h1>
  <span itemprop="author">Иван Петров</span>
</article>
```

### Сравнение подходов

| | JSON-LD | Microdata | RDFa |
|---|---------|-----------|------|
| Рекомендация Google | ✅ | ✅ | ✅ |
| Отделение от разметки | ✅ | ❌ | ❌ |
| Читаемость | Высокая | Средняя | Низкая |
| Динамическое обновление | Легко | Сложно | Сложно |
| Популярность | Высокая | Средняя | Низкая |

> 💬 **Вопрос на интервью:** «Почему JSON-LD предпочтительнее Microdata для SEO?»

**Ответ:** JSON-LD не зависит от HTML-структуры — можно добавить/изменить данные без правки разметки. Это критично при шаблонизации и A/B тестах. Кроме того, JSON-LD легче валидировать и поддерживать в больших командах.

## Вопросы на интервью

1. **В чём разница между `<strong>` и `<b>`? Когда использовать каждый?**
   > `<strong>` — семантическая важность, влияет на screen reader (интонация). `<b>` — стилистическое выделение без добавления важности (ключевые слова, продуктовые термины).

2. **Можно ли использовать `<article>` без заголовка?**
   > Технически можно, но WCAG рекомендует давать каждому `<article>` доступное имя (через `<h*>` или `aria-label`). Без имени screen reader объявит его просто как «article».

3. **Что такое implicit ARIA role и зачем это знать?**
   > Каждый HTML-элемент имеет встроенную роль в Accessibility Tree. `<button>` — `role="button"`, `<nav>` — `role="navigation"`. Добавлять явный `role` поверх нативного элемента — антипаттерн (`<button role="button">` — избыточно).

4. **`<dialog>` vs custom modal — в чём преимущества нативного элемента?**
   > Нативный `<dialog>` с `.showModal()` даёт: автоматический focus trap, ESC для закрытия, `::backdrop` псевдоэлемент, `top-layer` в браузере (не нужен z-index), правильные ARIA роли из коробки.

5. **Почему document outline algorithm важно знать, даже если он не работает?**
   > Кандидат должен знать историю: эта фича была в спеке, многие писали код под неё (множественные h1), и этот код до сих пор встречается в legacy. Понимание ловушки отличает Senior от Junior.
